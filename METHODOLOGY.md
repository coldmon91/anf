# 개발 방법론 — anf는 이렇게 만들어졌다

anf는 사람(제품 결정)과 AI(구현·측정·검증)가 짝으로 만든 앱입니다.
"바이브코딩이지만 성능은 역대급"이 가능했던 건 아래 규칙들 때문입니다.
전부 이 저장소에서 실제로 일어난 사례로 설명합니다.

## 1. 한 턴에 끝낸다 — 기본 루프

요청 하나가 들어오면 한 번의 작업 단위 안에서 끝까지 갑니다:

```
구현 → swift build → swift run anfTests → ./build.sh → 재실행(xattr+open) → git commit/push
```

"구현했어요, 확인해 보세요"로 끝나는 턴은 없습니다. 빌드가 깨졌거나 테스트가
빨간 채로 사용자에게 공을 넘기지 않습니다. CI(GitHub Actions)가 push마다
같은 검증을 한 번 더 돕니다.

## 2. 측정이 추측을 이긴다

성능 이슈는 **반드시 계측부터** 합니다. 이 저장소의 굵직한 수정은 전부
프로파일러가 범인을 지목했습니다:

- "정렬 눌렀더니 멈춤" → `sample`로 스택 확보 → `difference(from:)`(Myers diff)가
  26k 재정렬에서 O(N·D) 폭발 — 추측이었다면 정렬 알고리즘을 의심했을 것
- "진입이 느림" → 단계별 벤치 → 580ms 중 **정렬이 480ms** (벌크 읽기는 33ms로 무죄)
- "유휴인데 CPU 50%" → `sample` → 인덱스 스캔 폭주 발견
- 메모리 → `footprint` + `heap`으로 "라이브 힙 30MB vs 슬랙" 구분

도구: `sample`(행/스핀), `footprint`·`heap`(메모리), 자체 벤치 하니스(아래),
헤드리스 Chrome 스크린샷(랜딩 검증), AppleScript(실제 Finder 동작 계측).

## 3. 모든 성능 주장은 재현 가능한 하니스를 남긴다

수치를 말했으면 그 수치를 다시 뽑는 명령어가 저장소에 있어야 합니다:

| 하니스 | 명령 | 기준선 |
|---|---|---|
| 폴더 진입 | `ANF_BENCH=/폴더 swift run anfTests` | 26k 항목 ~210ms(debug) |
| 대용량 복사 | `ANF_BENCH_COPY=/폴더` | 55,860개/622MB ~7s (cp -Rc보다 빠를 것) |
| PDF 본문 스윕 | `ANF_BENCH_PDF=/폴더` | cold ~410ms / warm ~4ms |
| 전수 소크 | `ANF_SOAK=/루트` | 15만 디렉토리 무실패 |
| 행 감지 | `ANF_TRACE=1` 실행 | 메인 스레드 100ms+ 정지 자동 로깅 |

기준선은 TESTING.md에 기록 — 느려지면 사용자가 느끼기 전에 숫자가 먼저 어긋납니다.

## 4. 버그 신고는 회귀 테스트로 박제한다

사용자가 신고한 케이스를 **그 입력 그대로** 테스트로 만듭니다:

- "ㄱㅇㅇ 쳐도 금융위원회 안 나옴" → `금융위원회 × ㄱㅇㅇㅇㅎ/ㄱㅇㅇ` 테스트 추가
- 정렬 비치볼 → "26k 재정렬은 절대 diff 경로로 가면 안 됨 + 결정이 100ms 미만" 단언
- redo가 undo를 반복하던 버그 → FileUndo 왕복 테스트

규칙: **초선형(diff·정렬·매칭) 알고리즘은 최악 입력(전체 재정렬, 전부 불일치)으로
시간 상한을 단언하는 테스트가 필수.** 뷰 코드에 박힌 결정 로직은 순수 함수로
뽑아 단위 테스트한다(`ListDiff.strategy`가 표본).

## 5. 테스트는 4층 — 각 층이 다른 부류의 버그를 잡는다

1. **유닛/통합** (`swift run anfTests`, 자체 하니스 — CLT엔 XCTest가 없음):
   순수 로직 + 실제 `BrowserModel`을 픽스처 폴더에 물려 런루프 펌핑으로 구동
2. **UI 셀프테스트** (`ANF_UI_SELFTEST=1`): 합성 NSEvent를 `NSApp.sendEvent`로 쏴서
   로컬 이벤트 모니터·제스처의 **실제 경로**를 검증 — "유닛은 통과인데 실제 키는
   죽는" 부류(PgDn no-op 사건)는 이 층만 잡는다
3. **소크/벤치**: 규모에서만 터지는 문제 전담
4. **수동 체크리스트** (TESTING.md): 러버밴드, 실제 IME, 충돌 다이얼로그 등
   자동화 불가 항목 — 릴리즈 전 1회

환경 제약이 설계를 만들었습니다: 헤드리스 합성 키 입력은 이 앱에 닿지 않음 →
in-process 셀프테스트로 우회. CLT-only → 라이브러리(`anf`) + 얇은 실행 타깃
(`anfapp`) 분리로 `@testable` 확보.

## 6. 메인 스레드 불변식

버벅임 제로의 실체는 세 가지 금지였습니다:

- **블로킹 금지**: 동기 I/O·트리 워크·프로세스 실행은 전부 detached/background
  (복사 전 사이징, 팔레트 추출 스윕이 위반 사례였고 모두 추방)
- **키 입력당 O(n) 금지**: 26k 항목에서 키마다 도는 스캔은 캐시(자모 키, id→행 맵)로
  목록당 1회로 환원
- **UI 갱신 스로틀**: 진행률은 80ms 간격, 취소 플래그는 MainActor 홉 없이 락 기반

HUD는 바이트가 아니라 **시간 기준**(400ms 이상 살아 있는 작업만) — 작은 작업은
UI가 아예 안 뜨는 게 최고의 성능입니다.

## 7. 기준 동작은 실측하고, UX는 사용자가 이긴다

- Shift+화살표 논쟁: 진짜 Finder를 AppleScript로 계측해 "스네이크(누적)"임을
  확인 — 그런데 사용자는 그걸 싫어했고, **사각형 블록**으로 결정. 계측은
  선택지를 정확하게 만들 뿐, 최종 결정은 사용자의 몫
- 결정엔 근거를 남기고(왜 머티리얼이고 flat이 아닌지, 왜 핀≠Workspace인지)
  같은 논쟁을 반복하지 않는다

## 8. 릴리즈는 한 명령, 버전은 semver

- `./tools/release.sh <버전>`: 테스트 → 빌드 → zip → GitHub Release →
  Homebrew cask sha256까지 원커맨드
- semver: fix/docs만 → patch, feat 있으면 → minor (CONTRIBUTING.md에 명문화)
- 릴리즈마다 **brew 업그레이드를 실제로 돌려** `/Applications` 버전까지 확인
- 릴리즈 노트는 커밋 목록 복사가 아니라 사용자 언어로 다시 쓴다

## 9. 숫자는 정직하게, 한계도 같이

- 랜딩의 모든 수치는 위 하니스로 재현 가능 + 측정 환경(M4 Pro) 명시
- 한계도 적는다: 스캔본 PDF는 텍스트 레이어가 없어 검색 불가(OCR 안 함),
  self-signed라 Gatekeeper 안내 필요
- 다크패턴 제로: 텔레메트리 없음(네트워크는 하루 1회 버전 확인뿐), SSH 비밀번호는
  받지도 저장하지도 않음, 추적성 광고 문구 금지

## 10. 사람과 AI의 분업

- **사람**: 무엇을 만들지, 무엇이 이상한지(스크린샷 한 장이면 충분), 어느 UX가
  맞는지, 언제 출시할지
- **AI**: 구현, 계측, 회귀 박제, 배포 사이클, 문서 동기화 — 그리고 "안 된다"는
  신고를 의심하지 않고 재현부터 하는 것
- 신뢰의 단위는 말이 아니라 **돌려볼 수 있는 명령어**: 모든 주장 옆에 벤치,
  모든 수정 옆에 테스트, 모든 릴리즈 옆에 CI
